=============================
Django 1.0.2 リリースノート
=============================

:revision-up-to: 8961 (1.0)

Django 1.0.2 へようこそ!

これはDjang 1.0シリーズにおける２つ目の"バグフィックス"リリース で、
Django 1.0 コードベースの安定性とパフォーマンスを改善するものです。 
よってDjango 1.0.2 には新しい機能はふくまれていません (そして 
:doc:`我々の互換性ポリシー </misc/api-stability>` に 従って Django 1.0.0 
との上位互換性 を維持しています)が、多くのフィックスとと改善が含まれてい
ます。 Django1.0.2 は開発、あるいは本番で現在使っている、もしくはこれから
使おうとしている Django 1.0 をアップグレードするものとして推奨されます。 

Django 1.0.2 でのフィックスと改善
======================================

このリリースの主な理由は最後にリリースされた Django 1.0.1 での問題を解決
することです; Django 1.0.1 でつかわれたパッケージスクリプトでは最終リリース
パッケージから漏れたディレクトリ、 ``django.contrib.gis`` で必要とされる
ディレクトリとか、Djangoのユニットテストスイートの一部のディレクトリなど
が含まれています。

Django 1.0.2 ではパッケージスクリプトの更新が含まれていて、そのリリース
パッケージには Django 1.0.1 で漏れていたディレクトリが含まれています。
また、Django 1.0.1 からの全てのフィックスと改善を含んでいます;
詳細は :doc:`Django 1.0.1 リリースノート </releases/1.0.1>` を参照して
ください。


さらに Django 1.0.1 のリリースから、次のような問題が解決されました:

* ヘブライ語とデンマーク語の翻訳の更新が追加されました。

* Django モデルのデフォルトの ``__repr__`` メソッドは ``__unicode__`` 
  メソッドが正しくないユニコードデータを返した時でもより堅牢になりました。
  ;例外を返すのではなく、 ``repr()`` が不正なユニコードの箇所に 
  "[Bad Unicode Data]" 文字を含めるようになりました。
 
* Djangoの ``SafeUnicode`` クラスと MySQL アダプタの相互作用に関連したバグ
  が解決されました; ``SafeUnicode`` インスタンス(例えば、テンプレートレンダ
  リングで生成されます)がモデル属性に追加されて、明示的に ``unicode`` への
  中間的なキャストをしなくても MySQL に保存できるようになりました。

* SQLiteでは null可能な ``DateField`` をフィルタリングする際に生じるバグが
  解決されました。

* Django のドキュメントにいくつかの更新と改善が行われました。
